01
Script import MT5 → DuckDB #1 Python DuckDB
Script one-shot. Roule une seule fois.
  • Connexion MT5 via MetaTrader5 lib
  • Pull toutes les paires disponibles dans la watchlist MT5, tous TF (M1 M5 M15 H1 H4 D)
  • Chaque TF stocké séparément — zéro reconstruction en temps réel après
  • Table : candles(symbol VARCHAR, timeframe VARCHAR, time TIMESTAMPTZ, open DOUBLE, high DOUBLE, low DOUBLE, close DOUBLE, volume BIGINT)
  • PK + index composite : (symbol, timeframe, time)
  • Timestamps converties UTC→heure Québec offset-aware avant stockage
  • Validation : count par paire/TF, vérifier aucun gap majeur. Log final : X bougies, taille DB.
02
Création DuckDB #2 — Trades live DuckDB
DB persistante trades réels/démo.
  • Table : trades(id UUID PK, symbol VARCHAR, strategy_name VARCHAR, strategy_settings JSON, direction VARCHAR, entry_time TIMESTAMPTZ, entry_price DOUBLE, sl DOUBLE, tp DOUBLE, exit_time TIMESTAMPTZ, exit_price DOUBLE, profit_loss DOUBLE, risk_dollars DOUBLE, lot_size DOUBLE, ticket_id BIGINT, correlation_id UUID, exit_reason VARCHAR, conditions_met JSON, error_log VARCHAR)
  • Index : (symbol), (entry_time), (strategy_name)
  • Toutes timestamps TIMESTAMPTZ UTC offset-aware — affichage heure Québec
  • exit_reason : "TP" | "SL" | "ForceExit:[label]" | "OptionalExit:[label]"
  • conditions_met : JSON array des labels de conditions remplies au moment du signal
  • strategy_settings : snapshot JSON des Settings au moment du trade
03
DB Replay — Trades temporaires DuckDB
Fichier replay_trades.db. Même schéma exact que DuckDB #2.
  • Wipé à la fermeture de l'app
  • Wipé sur bouton Reset Replay
  • Recréé à chaque session — pas de migration
PHASE 02
PYTHON FASTAPI — MINI-BOT MT5
PYTHON
04
FastAPI mini-bot Python
Seul interlocuteur de MT5. C# ne touche jamais MT5 directement.
  • GET /status → {"python": bool, "mt5": bool} — timeout 3s
  • GET /account → {"balance", "equity", "drawdown_percent", "currency"} — timeout 5s
  • GET /candles?symbol&timeframe&count OU ?symbol&timeframe&from&to — timeout 10s
  • POST /order body: {"correlation_id": "uuid-v4", "symbol", "direction", "lot_size", "sl", "tp"} — timeout 10s
  • POST /close_order body: {"ticket": int} — timeout 10s
  • MT5 retourne UTC → converti heure Québec offset-aware AVANT envoi au C#
  • Toutes timestamps ISO 8601 offset-aware heure Québec ex: 2024-01-23T08:45:00-05:00
  • Prix 5 décimales. Lot size 2 décimales.
  • correlation_id déjà reçu → retourner ticket original sans renvoyer l'ordre
  • Ordre rejeté → 422 + reason. Marché fermé → 422 + "Market closed". Loggé dans DuckDB #2.
  • Pas de retry automatique sur aucun endpoint

PHASE 05
MOTEURS CORE
C#
11
EventBus C#
Bus central. Zéro couplage direct entre moteurs.
  • Subscribe<T>(handler) / Publish<T>(event)
  • Événements : NewCandleEvent, IndicatorsUpdatedEvent, TrendUpdatedEvent, TradeSignalEvent, TradeClosedEvent
12
IndicatorEngine C#
Calcule en continu sur tous les TF, toutes les paires.
  • Ichimoku complet : Tenkan, Kijun, Senkou A, Senkou B, Senkou A+26, Senkou B+26, Chikou
  • MACD : macd_line, signal_line, histogram
  • EMA 50, EMA 200
  • Calcul incrémental — jamais recalculer tout l'historique à chaque bougie
  • Résultats disponibles via GetIndicators(symbol, timeframe) → IndicatorValues
13
SwingPointEngine C#
  • Détecte HH/HL/LH/LL sur chaque TF en continu
  • N bougies de contexte = configurable par Strategy (dans Settings)
  • Résultats disponibles via GetSwingPoints(symbol, timeframe, count)
14
TrendEngine C# — Global, non paramétrable
Définition Bull/Bear/Range unique et non modifiable par Strategy.
  • Bull = EMA50 + EMA200 montent + structure HH/HL + prix au-dessus cloud
  • Bear = EMA50 + EMA200 descendent + structure LL/LH + prix sous cloud
  • Range = EMA50 ou EMA200 plate + compression LH/HL
  • Seuil slope "plate" = hardcodé dans TrendEngine
  • Résultat disponible via GetTrend(symbol, timeframe) → TrendState
  • Une Strategy peut ignorer TrendState — elle l'inclut juste pas dans ses conditions
15
RiskEngine C#
  • Input : capital actuel, RiskPercent (depuis Strategy.Settings["RiskPercent"] ou Strategy.RiskPercent), distance SL en pips
  • Output : lot_size (arrondi 2 décimales), risk_dollars
  • Vérifie MaxSimultaneousTrades avant de calculer
  • Vérifie MaxDailyDrawdownPercent — si dépassé, retourne null (aucun trade)
  • Zéro calcul de risk dans la Strategy
16
TradeManager C#
  • Entrée = open de la bougie SUIVANT le signal (jamais sur la bougie du signal)
  • Vérifie SL et TP à chaque bougie
  • Évalue ForceExitConditions → UNE vraie = ferme, exit_reason = "ForceExit:[label]"
  • Évalue OptionalExitConditions → UNE vraie + activée dans Settings = ferme, exit_reason = "OptionalExit:[label]"
  • Au redémarrage : TradeRepository.GetOpenTrades() + resync MT5 → reprend les trades ouverts
  • Identifie trades du bot via tag unique dans commentaire ordre MT5
  • Une position par paire par direction. Zéro hedging.
17
ExecutionManager C#
  • Live → MT5ApiClient.SendOrder() — market order uniquement
  • Replay → simule localement, sauvegarde dans DB Replay
  • Génère correlation_id UUID v4 avant chaque envoi
  • Rejet → loggé dans error_log du TradeRecord
  • Partial fills non supportés
18
StrategyRunner C#
Boucle principale. Ordre fixe et déterministe à chaque bougie.
  • Source → IndicatorEngine → SwingPointEngine → TrendEngine → Strategy.Evaluate() → RiskEngine → TradeManager → ExecutionManager
  • Replay : source = DuckDB (GetCandlesChunk), vitesse x1/x2/x4/x8
  • Live : source = MT5ApiClient (GetCandles), attend close de chaque bougie
  • Publie NewCandleEvent après chaque bougie pour l'UI
  • Source = DuckDB ou MT5 selon bouton pressé. Le moteur ne sait pas lequel.

PHASE 11
OPTIMIZER — MODULE SÉPARÉ DERNIÈRE PRIORITÉ
C#
35
OptimizableSettings — Extension de IStrategy C#
Chaque Setting peut avoir un mode fixe ou range.
  • OptimizableSettings : même structure que Settings mais chaque valeur peut être SettingValue (fixe) ou SettingRange(from, to, step)
  • Mode fixe → utilisé normalement en live et replay manuel
  • Mode range → utilisé par l'Optimizer pour générer les combinaisons
  • L'IA qui crée une Strategy doit préparer les deux en même temps
  • Tout paramètre logique de la Strategy doit être optimisable — ex: nombre de bougies sous le cloud, distance min Tenkan/Kijun, seuil MACD, etc.
36
OptimizerEngine C#
  • Génère toutes les combinaisons des ranges définis (brute force V1)
  • Lance un replay complet par combinaison via StrategyRunner
  • Multi-thread CPU — un thread par combinaison
  • Sauvegarde résultats dans DB Optimizer (schéma : combinaison + métriques)
  • Score multi-critères : net profit, max drawdown, profit factor, win rate, nb trades, avg RR, stabilité symboles
  • Pénalité si moins de 30 trades — combinaison rejetée
  • Période in-sample + out-of-sample — obligatoires
  • Walk-forward — V2
  • GPU compute (CUDA/OpenCL) pour parallélisation massive — V2
