2. Architecture
Python FastAPI — Mini-bot MT5
Seul interlocuteur de MT5. C# ne parle JAMAIS directement à MT5.
  • Endpoints : /status, /account, /candles, /order, /close_order
  • Toutes les timestamps retournées en heure Québec ISO 8601 offset-aware
  • MT5 retourne UTC → converti en heure Québec dans le FastAPI avant envoi au C#
StrategyLoader — Plug & Play via Roslyn VERROUILLÉ
Scanne /Strategies/ au démarrage. Compile les .cs via Microsoft.CodeAnalysis (Roslyn).
  • Nouvelle Strategy = glisser .cs + redémarrer. Zéro Visual Studio requis.
  • Apparaît automatiquement dans le dropdown Strategy
  • Erreur compilation → message dans UI, bot continue sans cette Strategy
  • Strategy retirée → disparaît au prochain redémarrage
StrategyRunner — Boucle principale
Ordre fixe et déterministe à chaque bougie :
Source → IndicatorEngine → SwingPointEngine → TrendEngine → Strategy.Evaluate() → RiskEngine → TradeManager → ExecutionManager
  • Replay : source = DuckDB, vitesse x1/x2/x4/x8
  • Live : source = MT5ApiClient, attend close de chaque bougie
EventBus
Bus central. Zéro couplage direct entre moteurs. Événements : NewCandleEvent, IndicatorsUpdatedEvent, TrendUpdatedEvent, TradeSignalEvent, TradeClosedEvent.
IndicatorEngine
Calcule en continu sur tous les TF, toutes les paires : Ichimoku complet, MACD, EMA50, EMA200. Calcul incrémental. La Strategy lit, ne recalcule jamais. Une Strategy NE PEUT PAS demander des indicateurs non fournis ici.
SwingPointEngine
Détecte HH/HL/LH/LL sur chaque TF en continu. N bougies de contexte = configurable par Strategy.
TrendEngine GLOBAL — NON PARAMÉTRABLE
Calcule Bull/Bear/Range sur tous les TF, toutes les paires. Définition unique, non modifiable par Strategy.
  • Bull = EMA50 + EMA200 montent + HH/HL + prix au-dessus cloud
  • Bear = EMA50 + EMA200 descendent + LL/LH + prix sous cloud
  • Range = EMA50 ou EMA200 plate + compression LH/HL
  • Seuil "plate" hardcodé dans TrendEngine
  • Une Strategy PEUT ignorer TrendState — elle l'inclut juste pas dans ses conditions
  • Une Strategy NE PEUT PAS modifier la logique Bull/Bear/Range
RiskEngine
Calcule lot_size depuis : capital actuel + RiskPercent (Strategy) + distance SL en pips. Vérifie MaxSimultaneousTrades et MaxDailyDrawdownPercent. Zéro calcul de risk dans la Strategy.
TradeManager
  • Entrée = open de la bougie SUIVANT le signal (jamais sur la bougie du signal)
  • Vérifie SL et TP à chaque bougie
  • Évalue ForceExitConditions → UNE vraie = ferme immédiatement
  • Évalue OptionalExitConditions → UNE vraie + activée dans Settings = ferme
  • Au redémarrage : resynchronise depuis MT5 + DuckDB #2. Trades ouverts repris.
  • Un trade appartient au bot via tag/commentaire unique injecté dans l'ordre MT5
  • Une position par paire par direction. Zéro hedging.
ExecutionManager
  • Live → MT5ApiClient.SendOrder() — market order uniquement
  • Replay → simule localement, sauvegarde dans DB Replay
  • Anti double-envoi : correlation_id UUID v4 par signal. Même ID déjà envoyé → ignore.
  • Partial fills : non supportés. Ordre rejeté = loggé, pas de retry.
  • Marché fermé ou ordre rejeté → loggé + alerte dans UI
3. IStrategy — Contrat final VERROUILLÉ
Règles absolues
Zéro méthode ShouldExit() — supprimée définitivement.
Toutes les sorties sont déclaratives via listes de conditions.
Une seule Strategy active à la fois. Settings ont priorité sur propriétés fixes si conflit.
Propriétés — liste complète finale
string Name Nom dans le dropdown
string Timeframe TF affiché dans le chart
List<string> RequiredTimeframes TF que le moteur doit calculer
List<string> Indicators Indicateurs affichés — valeurs : "Ichimoku" | "MACD" | "EMA50" | "EMA200"
decimal RiskPercent % capital par trade — surchargeable par Settings["RiskPercent"]
int MaxSimultaneousTrades Max trades ouverts simultanément toutes paires
decimal MaxDailyDrawdownPercent Bot stop si DD dépasse ce % dans la journée
StopLossRule StopLoss Type (AboveCloud|BelowCloud|SwingHigh|SwingLow|Fixed) + BufferPips
TakeProfitRule TakeProfit Type (RiskRatio|Fixed) + Ratio
List<StrategyCondition> LongConditions TOUTES vraies → entrer LONG
List<StrategyCondition> ShortConditions TOUTES vraies → entrer SHORT
List<StrategyCondition> ForceExitConditions UNE vraie → fermeture immédiate
List<StrategyCondition> OptionalExitConditions UNE vraie + activée Settings → fermeture
Dictionary<string,object> Settings Options UI — types valides : bool | decimal | int
StrategyCondition
string Label Affiché dans tooltip hover et popup trade
string Timeframe TF sur lequel évaluer
Func<IndicatorValues, TrendState, bool> Expression Lambda — reçoit valeurs moteur, retourne bool
Évaluée sur bougie courante. Bougie précédente = champ privé dans la Strategy.
4. Multi-strategy VERROUILLÉ
  • Une seule Strategy active à la fois — sélectionnée dans l'onglet Strategy
  • Plusieurs paires/indices tradées simultanément avec cette Strategy
  • Changement de Strategy s'applique au prochain Start uniquement

  • Chaque trade en DB lié à strategy_name + snapshot des Settings au moment du trade
5. API FastAPI — Contrat exact VERROUILLÉ
GET /status
Réponse 200 : {"python": true, "mt5": true}
MT5 down : {"python": true, "mt5": false} — Timeout : 3s. Appelé toutes les 5s.
GET /account
Réponse 200 : {"balance": 10000.00, "equity": 9850.00, "drawdown_percent": 1.5, "currency": "USD"}
MT5 down : 503 + {"error": "MT5 unavailable"} — Timeout : 5s.
GET /candles
Params : symbol, timeframe (M1|M5|M15|H1|H4|D), count OU from+to ISO 8601
Réponse 200 : [{"time": "2024-01-23T08:45:00-05:00", "open": 1.10500, "high": 1.10800, "low": 1.10400, "close": 1.10750, "volume": 1250}]
Timestamps heure Québec offset-aware. Prix 5 décimales. Volume entier. Timeout : 10s.
POST /order
Body : {"correlation_id": "uuid-v4", "symbol": "EURUSD", "direction": "BUY", "lot_size": 0.01, "sl": 1.10200, "tp": 1.11300}
Réponse 200 : {"ticket": 123456, "fill_price": 1.10750, "time": "2024-01-23T08:46:00-05:00"}
Rejeté : 422 + {"error": "Order rejected", "reason": "..."}
Marché fermé : 422 + {"error": "Market closed"}
correlation_id déjà reçu → 200 avec ticket original, pas de renvoi d'ordre.
Timeout : 10s. Pas de retry. Rejet loggé dans DuckDB #2.
POST /close_order
Body : {"ticket": 123456}
Réponse 200 : {"closed": true, "close_price": 1.11300, "time": "2024-01-23T09:15:00-05:00"}
Ticket introuvable : 404 + {"error": "Ticket not found"} — Timeout : 10s.
Règles globales
  • Toutes les dates ISO 8601 offset-aware heure Québec
  • Prix arrondis à 5 décimales. Lot size à 2 décimales.
  • Chaque ordre a un correlation_id UUID v4 généré côté C#
  • Python down → bouton "Start Trading" grisé + message UI
6. Schémas DB VERROUILLÉ
DuckDB #1 — Candles historiques (lecture seule)
candles(symbol VARCHAR, timeframe VARCHAR, time TIMESTAMPTZ, open DOUBLE, high DOUBLE, low DOUBLE, close DOUBLE, volume BIGINT)
PK : (symbol, timeframe, time) — Index composite identique.
Tous les TF stockés séparément à l'import. Zéro reconstruction en temps réel.
DuckDB #2 — Trades live (persistant)
trades(id UUID PK, symbol VARCHAR, strategy_name VARCHAR, strategy_settings JSON, direction VARCHAR, entry_time TIMESTAMPTZ, entry_price DOUBLE, sl DOUBLE, tp DOUBLE, exit_time TIMESTAMPTZ, exit_price DOUBLE, profit_loss DOUBLE, risk_dollars DOUBLE, lot_size DOUBLE, ticket_id BIGINT, correlation_id UUID, exit_reason VARCHAR, conditions_met JSON, error_log VARCHAR)
Index : (symbol), (entry_time), (strategy_name)
strategy_settings : snapshot JSON des Settings au moment du trade
conditions_met : liste JSON des labels de conditions remplies au moment du signal
exit_reason : "TP" | "SL" | "ForceExit:[label]" | "OptionalExit:[label]"
error_log : null si succès, message si erreur
DB Replay — Trades temporaires
Même schéma que DuckDB #2. Fichier replay_trades.db.
Wipé à fermeture app et sur bouton Reset Replay. Recréé à chaque session.

10. GPU & Performance VERROUILLÉ
🔵 Niveau 1 — GPU UI (Obligatoire — V1)
Objectif : Replay ultra fluide. Priorité #1 du bot.
Rendu chart via WebView2 (Chromium GPU).
  • Virtualisation des données — afficher seulement les bougies visibles à l'écran
  • Chunk loading intelligent — charger par blocs de 500–2000 bougies
  • Zéro rerender complet — update uniquement la dernière bougie
  • Buffer mémoire propre — cache des bougies déjà chargées
  • Throttling UI — max 60 FPS
Résultat attendu : scroll fluide, replay x1→x8 sans lag, changement symbol/timeframe instantané.
🟢 Niveau 2 — CPU Scale (Préparé — V1/V2)
Objectif : Supporter beaucoup de données sans casser le replay. CPU optimisé uniquement.
  • Calcul incrémental des indicateurs — jamais recalcul complet
  • Cache par symbole + timeframe
  • Multi-thread : thread moteur | thread UI | thread data loader — séparés
  • Lecture DuckDB par chunks — jamais full load en mémoire
  • Préchargement intelligent — buffer adaptatif replay (x1=60 | x2=120 | x4=240 | x8=480)
Résultat attendu : 10–50 symboles sans crash, plusieurs TF simultanés, replay accéléré stable.
11. Optimizer DERNIÈRE PRIORITÉ
Principe
Module complètement séparé du bot principal. Le bot reste clean. L'optimizer utilise le même moteur, les mêmes Strategy, les mêmes données DuckDB.
Comment ça fonctionne
  • Chaque Setting d'une Strategy peut avoir un mode valeur fixe ou range (de / à / pas)
  • L'optimizer génère toutes les combinaisons des ranges définis
  • Lance un replay/backtest complet par combinaison
  • Sauvegarde et classe les résultats selon un score multi-critères
  • V1 : brute force multi-thread CPU — V2 : GPU compute (CUDA/OpenCL)
Score multi-critères (pas juste le profit)
  • Net profit
  • Max drawdown
  • Profit factor
  • Win rate
  • Nombre de trades (pénalité si moins de 30 — pas assez de données)
  • Average RR
  • Stabilité entre symboles sélectionnés pour le run
Garde-fous anti-overfitting
  • Période in-sample (entraînement) + période out-of-sample (validation) — obligatoires
  • Walk-forward validation — V2
  • Pénalité combinaisons avec trop peu de trades
  • Score calculé sur les paires sélectionnées pour ce run uniquement
UI — Onglet Optimizer
  • Sélecteur Strategy
  • Pour chaque Setting optimisable : checkbox "Optimiser" + champ De + champ À + champ Pas
  • Sélecteur symboles + période (in-sample + out-of-sample)
  • Bouton Start Optimizer
  • Tableau résultats classés : toutes les métriques par combinaison
  • Chaque ligne = une combinaison testée — affiche les métriques ET les valeurs exactes des settings qui ont produit ce résultat
  • Colonnes : score global, net profit, max drawdown, profit factor, win rate, nb trades, avg RR — + une colonne par setting optimisé avec sa valeur
  • Bouton "Appliquer cette combinaison" → injecte les valeurs dans la Strategy
